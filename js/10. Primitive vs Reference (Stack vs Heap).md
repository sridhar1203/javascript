# Primitive vs Reference – Stack vs Heap

## Main idea

- **Primitive types** → stored **by value** → typically in **stack** memory.
- **Reference types** (objects, arrays, functions) → actual data in **heap**; variable in **stack** holds a **reference** (address) to that heap location.

So the main concept is: **where and how** data is stored – **stack** (fixed, fast, for values and references) vs **heap** (larger, for big data; we keep only a “pointer” in the stack).

### Stack vs Heap – difference at a glance

| | **Stack** | **Heap** |
|---|-----------|----------|
| **Stores** | Primitive values (number, string, boolean, etc.) and references (addresses/pointers) | Actual content of objects, arrays, and other reference-type data |
| **Size** | Limited, fixed-size slots | Larger, dynamic (can grow) |
| **Speed** | Fast (direct access) | Slower (access via reference from stack) |
| **Lifetime** | Short-lived; tied to function/block scope (cleared when scope ends) | Lives until no reference points to it; then garbage collected |
| **Order / structure** | Last In, First Out (LIFO); ordered | No fixed order; managed by the engine |
| **What lives here** | Local variables, function arguments, return addresses, references (pointers) | Objects, arrays, function bodies (as objects), etc. |
| **Copy behaviour** | Copying a variable copies the value (or the reference value); primitives get a new value | Copying a variable copies only the reference; both point to the same heap data |

---

## Stack memory

- **What:** A region of memory that stores **local variables** and **function call info**.
- **How it works:** Fixed-size slots; **last in, first out** (like a stack of plates). When a function ends, its variables are removed from the stack.
- **Stores:** Primitive **values** (number, string, boolean, etc.) and **references** (the address that points to heap). Not the full object/array.
- **Characteristics:** Fast, limited size, short-lived (tied to function scope).

---

## Heap memory

- **What:** A region of memory for **dynamic**, larger data.
- **How it works:** Objects, arrays, and other reference-type values live here. They can grow and stay as long as something references them.
- **Stores:** The **actual content** of objects and arrays (properties, elements).
- **Characteristics:** Larger space, slightly slower to use (we go through the reference), managed by the engine (garbage collection when nothing points to it).

---

## Primitive types (stored in stack – by value)

In JavaScript, **primitives** are: `string`, `number`, `boolean`, `undefined`, `null`, `symbol`, `bigint`.

- The **value itself** is stored in the stack (or in a fixed-size slot the engine uses for primitives).
- When you **copy** a primitive, you get a **new copy** of the value. The two variables are independent.

```javascript
let a = 10;
let b = a;   // copy the value: b gets 10
b = 20;      // only b changes
console.log(a);  // 10  (unchanged)
console.log(b);  // 20
```

**In memory (concept):**
- Stack: `a` → `10`  
- After `let b = a`: stack: `a` → `10`, `b` → `10` (two separate values)  
- After `b = 20`: `a` → `10`, `b` → `20`

---

## Reference types (object in heap, reference in stack)

**Reference types** in JS: `object`, `array`, `function` (and others that are objects under the hood).

- The **actual data** (object/array) is stored in the **heap**.
- The variable in the **stack** holds only a **reference** (address/pointer) to that heap location.
- When you **copy** the variable, you copy the **reference**, not the full object. So both variables **point to the same** object in the heap. Changing the object through one variable is visible through the other.

```javascript
let obj1 = { name: "Ali" };
let obj2 = obj1;   // copy the reference, not the object
obj2.name = "Bella";
console.log(obj1.name);  // "Bella"  (same object!)
```

**In memory (concept):**
- Stack: `obj1` → **ref** (e.g. address 0x001)  
- Heap: `{ name: "Ali" }`  
- After `let obj2 = obj1`: stack: `obj1` → **ref**, `obj2` → **same ref** (both point to same heap object)  
- After `obj2.name = "Bella"`: heap object becomes `{ name: "Bella" }`; both `obj1` and `obj2` still point to it, so `obj1.name` is also `"Bella"`.

---

## Quick comparison

| | Primitive | Reference (object, array, function) |
|---|-----------|--------------------------------------|
| Stored where | Value in **stack** (conceptually) | **Object** in **heap**; **reference** in **stack** |
| Copy means | Copy the **value** (new copy) | Copy the **reference** (same object) |
| Mutate “copy” | Other variable unchanged | Other variable “sees” the change (same object) |
| Examples | `string`, `number`, `boolean`, `undefined`, `null`, `symbol`, `bigint` | `object`, `array`, `function` |

---

## Why this matters

1. **Copying:** Primitives → real copy. References → only the address is copied, so you have two names for one object.
2. **Comparison:** Primitives compared by value (`5 === 5`). References compared by reference (`{} === {}` is `false` – two different objects in heap).
3. **Mutation:** Changing a property of an object affects every variable that holds a reference to that object. Primitives don’t have “shared” state like that.

---

## Summary

| Term | Meaning |
|------|--------|
| **Stack** | Stores primitive **values** and **references** (addresses). Fast, limited, tied to scope. |
| **Heap** | Stores the **actual** objects/arrays. Variables hold a reference (pointer) in the stack to this data. |
| **Primitive** | Stored by value (in stack). Copy = new value. Types: string, number, boolean, undefined, null, symbol, bigint. |
| **Reference type** | Data in heap; variable in stack holds reference. Copy = same object. Types: object, array, function, etc. |

**Mnemonic:** **Primitive** = value in stack, copy is independent. **Reference** = object in heap, variable holds “address”; copying the variable copies the address, so both point to the same place.
